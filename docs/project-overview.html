1 Overview

Personal Workplanner (package name personal-workplanner) is a single-page web app for planning your work week and comparing intent against reality. You lay out the week's tasks, track actual time on each one with start/stop timers, flag unplanned interruptions as they happen, and review the gap between what you planned and what you actually did.

  • Plan the week — group tasks into sections and give each an hour estimate.
  • Track actual time — a live start/stop timer accumulates real seconds per task.
  • Flag interruptions — mark work as unplanned to separate deliberate work from surprises.
  • See the gap — metric tiles and a review view surface planned load vs. actual tracked, completion rate, and unplanned hours.

Architecture at a glance

Front endReact 18 + TypeScript, built with Vite.
Back endNetlify Functions + Netlify Blobs for storage, with Netlify Identity for authenticated, per-user persistence.
Persistence modelEvery signed-in user gets their own isolated dataset, stored server-side under a key derived from their user id.

2 File / Directory Structure

3 Key Data Objects & Relationships

Everything hangs off one root object: a WeekPlan. There is exactly one per user's current week. It carries the owner's profile, capacity, review notes, and two collections: sections (WorkSection[]) and tasks (WorkTask[]). All types below are taken verbatim from src/types.ts.

WeekPlan — the root object

id: string
Identifier, e.g. week-2026-07-13.
ownerName: string
Display name shown in the header.
ownerRole: string
Role label (e.g. "Application Coordinator").
weekStart: string
ISO date (YYYY-MM-DD) of the Monday the week begins.
weeklyCapacityHours: number
Planning budget; drives the "capacity remaining / over capacity" metric.

Free-text weekly summary edited on the Review tab.
sections: WorkSection[]
The groupings tasks belong to.
tasks: WorkTask[]
Every task for the week (flat list, not nested under sections).
updatedAt: string
ISO timestamp; re-stamped on every mutation via setPlan.

WorkSection — a grouping of work

id: string
Referenced by tasks through sectionId.
name: string
e.g. "Project Work", "Support & Follow-up".
focus: string
Short description of what the section groups.
color: string
Hex color used for the task accent stripe (e.g. #1f6feb).

WorkTask — a single unit of work

Unique id (prefixed task-).
title: string
Task or outcome.
sectionId: string
Reference to its WorkSection.id — a string ref, not a nested object.

Which day it belongs to: "mon" | "tue" | "wed" | "thu" | "fri", or "backlog" (unscheduled).

The plan — how long you expect it to take.

Separates deliberate work (true) from interruptions (false).

The reality — accumulated tracked time (excludes the currently running interval).

   estimateHours  = the PLAN            actualSeconds + timerStartedAt = REALITY
   planned: bool  = deliberate work (true) vs. interruption (false)
  • A WorkTask references its section by sectionId — a plain string ref, not an embedded object. The UI resolves it with getSection(sectionId, sections).
  • A task belongs to exactly one day slot: "mon".."fri" or "backlog".
  • estimateHours is the plan; actualSeconds + timerStartedAt together are reality; planned splits intent from interruption.

Timer math

Displayed time for a task combines saved seconds with the live running interval:

// from App.tsx
secondsForTask(task, now)
    = task.actualSeconds
    + (task.timerStartedAt
         ? floor((now - timeOf(task.timerStartedAt)) / 1000)
         : 0)
  • A 1-second setInterval bumps a now state value (Date.now()) every second, which re-renders all live timers without touching the plan.
  • Starting one timer stops any other running task. startTimer maps over every task calling stopRunningTask (which folds the elapsed run into actualSeconds and clears timerStartedAt), then sets the chosen task to status: "active" with a fresh timerStartedAt.

4 Data Flow & Persistence (3 layers)

State lives in one place and flows outward through three layers. The single source of truth is one WeekPlan held in React state inside the usePlanPersistence hook. It is mutated only through setPlan, which re-stamps updatedAt on every change. 

  Layer 1: REACT STATE                Layer 2: BROWSER            Layer 3: SERVER
  ─────────────────────               ──────────────             ────────────────
  usePlanPersistence()
    one WeekPlan in useState
        │
     setPlan(updater)  ── re-stamps updatedAt
        │
        ├──────────────►  localStorage           (instant, always)
        │                 key "personal-workplanner:plan"
        │
        └── debounce ~700ms ──► POST /api/workplan  ──►  netlify/functions/workplan.ts
                                { plan }                     │
                                                             ├─ getUser()  (Netlify Identity)
                                                             │     └─ no user? 401
                                                             │
                                                             └─ getStore("workplan")
                                                                    .setJSON("plan:<user.id>", plan)
                                                                            │
                                                                            ▼
                                                                     Netlify Blobs

  ON LOAD:  loadLocalPlan()  ─► render immediately
            then loadRemotePlan() (GET) ─► if a remote plan exists, overlay it & re-save local

Layer 1 — React state

usePlanPersistence owns the plan and a SyncState. Two effects react to plan changes: one writes to localStorage synchronously, the other schedules the debounced remote save.

Layer 2 — localStorage

On every plan change, saveLocalPlan writes JSON to localStorage["personal-workplanner:plan"] immediately — so the app is fully usable and durable offline. On startup loadLocalPlan reads local first (falling back to createDefaultPlan()), so the UI paints instantly before the network resolves.

Layer 3 — serverless function + Netlify Blobs

A change also schedules a ~700ms debounced POST to /api/workplan (each new keystroke clears the pending timer). The request lands on netlify/functions/workplan.ts, which:

  • calls getUser() (Netlify Identity) and returns 401 if unauthenticated;
  • opens the blob store getStore({ name: "workplan", consistency: "strong" });
  • stores/reads JSON under the key plan:<user.id>.
That per-user key — plan:<user.id> — is why every account gets an isolated dataset. GET returns { plan } (or null); POST validates the payload then setJSONs it. Any other method returns 405.

Load ordering & the sync state machine

On load the app reads local first, then overlays the remote plan if the GET succeeds and returns one (and re-saves it locally). The SyncBadge in the header reflects the SyncState machine:

local loading synced saving offline error

  • local → initial, before any remote check.
  • loading → the startup GET is in flight.
  • synced → remote reachable / last save succeeded.
  • saving → a debounced POST is in flight.
  • offline → the startup GET failed (remote unavailable; local still works).
  • error → a save POST failed.

Remote saves are gated on remoteAvailable — the app only starts POSTing after it has confirmed the backend is reachable, so a purely local/offline session never thrashes the network.

Routing note: netlify.toml redirects /api/workplan → /.netlify/functions/workplan (status 200), so the client's tidy /api/workplan URL maps onto the actual function endpoint.

5 Page Design / Component Tree

There is no routing. <App> gates on authentication: while the session is resolving it shows a loading panel; with no user it renders <AuthGate> (sign-in / register); once authenticated it renders <PlannerApp>.

Inside PlannerApp the header carries the identity block, a Planner ⇆ Manager segmented control (the WorkView switch), the SyncBadge, a sign-out button, and a settings toggle. Below it sits an optional settings band, then a week hero and a 4-tile metrics grid. The body then switches on the current view. 

<App>                                   // auth gate — no router
│
├── (auth loading)  ─────────────────  loading panel
├── (no user)       ─────────────────  <AuthGate>         // sign-in / register form
│
└── (signed in) <PlannerApp>
    ├── <header>
    │     ├── identity (name, email)
    │     ├── segmented control  ──── WorkView: planner ⇄ manager
    │     ├── <SyncBadge>
    │     ├── sign-out button
    │     └── settings toggle
    ├── settings band            // name, role, capacity, week start, "start blank week"
    ├── week hero                // week label + last-updated stamp
    ├── metrics grid (×4 tiles)  // derived via useMemo from the plan + now
    │
    ├── view === "planner":
    │     ├── tab bar  ──── PlannerTab: daily | weekly | review
    │     ├── <DailyView>   ── day picker + <TaskForm> + Today / Unscheduled columns of <TaskCard>
    │     ├── <WeeklyView>  ── <TaskForm> + section editor + per-section lists of <TaskCard>
    │     └── <ReviewView>  ── <ReviewPanels> + weekly-summary textarea
    │

          └── <ManagerView> (read-only)  ── <ReviewPanels> + per-day rollup + summary

   Reused building blocks:
     <TaskCard>     task row: timers, status, day select, est / actual / gap  (Daily + Weekly)
     <TaskForm>     add-task form: title, section, day, estimate, unplanned flag
     <ReviewPanels> Planned / Completed / Unexpected / Time-gaps  (shared by Review + Manager)
     <EmptyState>   placeholder for empty lists

Two nav levels

Navigation is entirely state-driven, with two independent switches:

  • WorkView — planner vs. manager (the header segmented control).
  • PlannerTab — daily / weekly / review (the tab bar, only shown in planner view).

The 4 metric tiles

Derived with useMemo from the plan and now:

TileShows
Planned loadSum of estimateHours over planned tasks, vs. weekly capacity.
Actual trackedTotal tracked hours; tone flips to red when over capacity.
Planned completedCompletion rate = done planned estimate / total planned estimate.
Unplanned workHours logged on planned:false tasks + count.

Shared review panels

ReviewPanels renders four ReviewPanel cards — Planned, Completed, Unexpected, and Time gaps (tasks running more than ~0.25h over estimate). It is used by both the Review tab and the read-only Manager view.

State is intentionally simple: one WeekPlan, no Redux and no React context. Data flows down as props and mutations flow up through callback props (startTimer, stopTimer, setTaskStatus, deleteTask, setPlan, …), all ultimately routed through setPlan.
